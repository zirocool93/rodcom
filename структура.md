13. Финансовый журнал

Это очень важная часть. Балансы лучше считать не напрямую из платежей и расходов, а через журнал.

ledger_entries
ledger_entries
- id UUID
- school_id UUID
- class_group_id UUID
- account_id UUID
- campaign_id UUID NULL
- family_id UUID NULL
- source_type VARCHAR
- source_id UUID
- direction VARCHAR
- amount_cents BIGINT
- currency VARCHAR DEFAULT 'RUB'
- entry_date TIMESTAMP
- description TEXT NULL

source_type:

payment
expense
refund
transfer
manual_adjustment

direction:

in
out

Пример:

Payment confirmed:
direction = in
amount = 120000
source_type = payment

Expense paid:
direction = out
amount = 350000
source_type = expense

Тогда баланс считается так:

SUM(in) - SUM(out)

И отчеты становятся проще.

49. Важная бизнес-логика
Подтверждение платежа

Когда казначей подтверждает платеж:

payments.status = confirmed
создается запись в ledger_entries
обновляется payment_allocations
обновляется payment_requests.amount_paid_cents
обновляется статус участника сбора
создается уведомление родителю
пишется audit_log
Проведение расхода

Когда расход отмечен как оплаченный:

expenses.status = paid
создается запись в ledger_entries с direction = out
уменьшается баланс сбора/класса
прикрепляется чек, если есть
создается уведомление
пишется audit_log
Генерация отчета

Когда формируется отчет:

система берет ledger_entries
группирует по сбору/категориям/датам
подтягивает расходы и чеки
формирует data_json
создает PDF/XLSX
сохраняет файлы
меняет статус отчета на generated
